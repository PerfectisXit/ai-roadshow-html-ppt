## 3. 单文件 HTML 的内部结构

`ai-roadshow.html` 是一个完整的单页 PPT 应用，内部大致分成 5 块：

### A. 全局样式区

文件开头到 `const brandIcon` 之前，基本都是 CSS。

重要内容：

- 主题变量：`:root`
- 画幅和舞台：`#app`、`#deck-shell`、`#deck`
- 幻灯片基础结构：`.slide`、`.slide-header`、`.slide-body`、`.body-wrap`
- 通用卡片体系：`.card`、`.proof-card`、`.rail-card`、`.quote-card`、`.roadmap-card`
- 布局模板：`.statement-shell`、`.focus-stage`、`.scenario-board`、`.governance-board`
- 组件类：`.micro-flow`、`.token-row`、`.feature-pair`、`.decision-mini-grid`、`.warning-card`、`.redline`

### B. 品牌资源

检索关键词：

- `const brandIcon = {`

这里集中定义品牌 logo 的资源地址。

### C. slides 数据区

检索关键词：

- `const slides = [`

这是整份 PPT 的核心内容数据源。

每页基本长这样：

```js
{
  id: "s39",
  layout: "strategy",
  kicker: "...",
  title: "...",
  subtitle: "...",
  html: `...`
}
```

重要字段：

- `id`: 稳定页面标识，后续定位、review、修样式优先用它
- `titleText`: 由脚本生成的纯文本标题，供浏览器标题和定位逻辑使用
- `layout`: 页面布局模板
- `className`: 额外样式控制，比如 `micro`
- `noHeader` / `noPage`: 章节页常用
- `html`: 当前页主体 DOM

### D. 页面规范化与定位逻辑

检索关键词：

- `const SLIDE_KICKERS =`
- `const SLIDE_TITLE_PREFIXES =`
- `const SLIDE_SUBTITLE_OVERRIDES =`
- `const SLIDE_CLASS_FIXES =`
- `function normalizeSlides()`
- `function assignPageMetadata()`
- `function findSlideIndexByRef(raw)`

这是后期维护最重要的一层。它负责：

- 统一标题前缀和副标题清理
- 统一 slide 额外 class
- 生成纯文本标题 `titleText`
- 生成页码相关元数据
- 统一页面定位规则

现在页面有 3 套明确身份：

- `slide id`
  - 稳定真源，例如 `s10`、`s37`、`s47`
- `absolute page`
  - `slides` 数组中的绝对顺序
- `visible page`
  - 真正显示在右上角页码里的顺序，只统计 `noPage !== true` 的页面

DOM 上对应：

- `data-slide-id`
- `data-absolute-page`
- `data-visible-page`

定位规则：

- 改代码、写样式、做 review 时，优先用 `slide id`
- 跟用户对齐“第几页”时，再结合 `visible page`
- 不要再手写“第 11 页大概对应 s10”这种临时映射

### E. 渲染与动画逻辑

检索关键词：

- `function renderDeck()`
- `function renderSlideMarkup(slide, index)`
- `function renderSlideHeader(slide)`
- `function renderSlideBody(slide)`
- `function renderSlidePagePill(slide)`
- `function fitDeck()`
- `function showSlide(index)`
- `function runAnimations(slide, token)`

这几块负责：

- 把 `slides` 数组渲染成真正的 DOM
- 根据窗口缩放到 16:9
- 控制翻页和 hash
- 为当前页播放动画

维护建议：

- 后续继续整理渲染层时，优先往小的辅助函数和注册表方向拆
- 不要再把新的结构判断塞回一个超长 `renderDeck()` 模板里

## 8. 当前已验证有效的设计方向

这是后续修改时必须延续的方向。

### 修改前置流程

#### 章节级前置流程

对某一章节里的页面开始修改之前，不要立刻逐页开改。必须先把这一整章全部过一遍，形成章节级判断：

- 先看完整章一共要讲哪些内容
- 先看这一章当前分页是否合理
- 先判断哪些页信息太少、可以合并
- 先判断哪些页信息过满、应该拆开
- 先判断页面顺序是否符合讲演节奏
- 先判断这一章的情绪曲线、说服链路和重点转折是否清楚

也就是说：

- 先完全掌握这一章要讲的全部内容
- 再构思这一章应该怎么分页
- 再决定是否要做页面合并、拆分、顺序调整
- 最后才进入单页的设计和精修

章节级思考的目标仍然只有一个：

- 服务讲演效果
- 服务叙事节奏
- 服务观众理解

章节级执行顺序固定为：

1. 重看该章节在 `路演思路.md` 中的原始内容
2. 重看 README 和母版里的当前规则
3. 看当前实现和现有截图
4. 判断这章是否要合并页、拆分页、调整顺序
5. 先定这章的高潮页、转折页、收束页
6. 再进入单页修改

#### 章节开改前必须先回答的 5 个问题

- 这一章的核心任务是什么？是立观点、讲案例、讲方法、讲边界，还是讲方案？
- 这一章现在的分页，是否真的有利于演讲，而不是只是按写作习惯切段？
- 有没有两页讲的是同一件事、但被不必要地拆开了？
- 有没有一页承载了太多信息，观众一眼看不清主次，需要拆分？
- 这一章里哪几页应该是高潮页、转折页、收束页？它们现在是否足够突出？

对任何一页做重新设计或精修之前，不要直接开始排版。必须先完成这一步脑内判断：

- 先模拟真实演示场景，判断演讲者这页真正要突出讲什么
- 先判断观众在这一页最应该被吸引去看哪里
- 先确定这一页的视觉重心应该是什么
- 先判断哪些内容是主信息，哪些只是辅助信息，哪些其实可以删掉
- 先判断原页里哪些措辞可以优化得更利于讲演，哪些句子太书面、太长、太散
- 先判断哪些文字没有实际作用，只是在“占地方”，应删掉或并入别处
- 先判断这页是否需要增加动作型组件、动画、图示或更强的排版层级来辅助讲演
- 先判断这页真正需要保留多少文字
  - 如果三句话就够，就只留三句话
  - 不允许为了排版规整而把说明文字越补越多

一句话原则：

- 不是为了设计而设计
- 不是为了页面饱满而硬塞内容
- 一切都应服务于讲演效果
- 一切都应服务于把观众的注意力放到演讲者真正想让观众看的地方

#### 每页开改前必须先回答的 6 个问题

- 这页的主句是什么？观众离开这一页后最应该记住什么？
- 这页的视觉重心应该落在哪个模块？
- 讲演时演讲者会先讲哪里、再讲哪里？页面视线顺序是否一致？
- 原页面哪些文字可以删，哪些可以压缩，哪些必须强化？
- 这页有没有任何“只对设计者 / 讲者 / AI 自己有意义”的解释词、备注词、分镜词、标注词？如果有，必须只留在代码注释里，绝不能上屏。
- 这页有没有为了撑满版面而出现的重复信息？同一个原因、同一个结论、同一个标签，不允许在同一页左右两侧各说一遍。
- 是应该通过重排、删减、提炼来变强，还是应该补一个更有说明力的动作型组件？
- 如果要补组件，它是在解释“发生了什么”、还是在证明“为什么可信”、还是在帮助“快速收束”？

单页级执行顺序固定为：

1. 先想讲演目标
2. 再定视觉重心
3. 再删无效文字、压缩拖沓措辞
4. 再决定版式和组件
5. 最后才是具体排版和动画

### 有效方向

- 不要只把卡片拉高，要把卡片内部做出层级
- 大块留白优先换成“有展示意义”的小组件
- 优先做“动作型组件”，而不是继续堆说明文字
- 动效组件、动画组件、拟真组件的优先级要明显提高
  - 只要它能更好地表达、解释或强化记忆，就优先考虑用组件而不是补文字
  - 理想状态下，大多数页面都应至少有一个形式有趣且有意义的动效/动画组件
- 图标、logo、示意插画、内嵌 SVG 也应作为高优先级手段参与页面表达和消化留白
  - 讲到 OpenAI / ChatGPT / OpenClaw / GitHub 等具体对象时，应在合适位置放标准 logo
  - 讲到“眼睛”“手”“大脑”“扫描”“命中”“归并”等抽象概念时，应尽量做成图标化或图形化表达
  - 优先使用标准素材库可复用 logo；如果没有合适素材，可以直接手写 SVG / data URI 内嵌到 HTML 中
  - 不要为了单文件 HTML 去外挂本地图片；图片、图标、插画都应尽量以内嵌代码形式存在
- 主组件不是越复杂越好，而是要更贴近讲演者这一页真正想强调的意图
  - 如果组件很花但和这页核心意思关系不大，也是错的
  - 如果组件形式简单，但能准确承接主句、问题或判断，反而更好
- 如果某一侧文字天然不多，不要硬把说明卡放大撑高；优先改成能占画幅的动作型组件板，例如“翻译流动示意”“标签筛选动画”“扫描/检索/命中过程板”
- 让页面像“正在发生的演示现场”
- 用 `主舞台 + 辅助说明 + 收束条` 的结构组织页面
- 图示、状态灯、流线、命中、扫描、追溯、步骤激活都优于单纯大段落
- 先确定讲演目标，再决定版式和组件，而不是先做版式再往里塞内容
- 页面里的每一句话都要有讲演作用；无意义的解释、重复说明、占地方的文字要主动删
- 能用更短、更准、更利于口播的措辞表达，就不要保留冗长书面句
- 动画、组件、卡片的存在都必须有讲演意义：要么帮助理解，要么帮助证明，要么帮助收束
- 任何只对创作者有意义的话，绝不能显示在观众页面里
  - 例如“先让观众想起自己”“这里是开场基调”“下面要抬出重点”“这一页要讲什么”这类话，只能写成代码注释，不能作为页面文案
- 同一页内不能为了饱满而重复表达同一层意思
  - 如果“工具刚成熟 / 方法未锁死 / 经验最值钱”已经在一侧完整成立，另一侧就不应再用另一种说法重复一遍
- 母版固定的是视觉底板，不是内容结构
- 双栏不是硬规则；单栏、大字主句页、问题页、满屏主组件页都应该被优先考虑
- 双栏比例、是否双栏、右侧是否拆上下模块，都要跟着页面内容和讲演逻辑走
- 如果某页本来就没有太多该说的文字，不要硬做双栏，也不要为了守母版去补次要卡片
- 可以直接用“少量重文字 + 超大有意义动效组件”的结构解决留白
- 发现留白后，先判断空白区域的面积、形状和位置，再决定是放大主舞台、重做组件，还是补一个有演示意义的模块
- 填补留白优先遵守“七巧板思路”：先看现有主块能否自然长大，其次才是新增组件；新增组件必须与页面主叙事有关
- 如果一页仍然显得空，不只盯留白本身，还要判断两件事：
  - 是不是主组件本身还不够贴近讲演意图，所以虽然占了位置却不够“成立”
  - 是不是应该让其他信息块更充实，从而压缩当前容器的无效空高
- 如果某页内容本来就少，但这页仍然值得单独存在：
  - 先判断这页是否已经足够重，不要先想着补文字
  - 优先放大真正重要的那一句话或那一个主结论
  - 再根据留白的面积和形状，补一个有意义的大型动作型组件
  - 组件必须服务表达，可以是流程示意、可视化解释、命中/扫描/归并/映射等，但不能只是为了占地方
  - 能只保留灵魂三问、一个判断、一个强结论，就不要为了版面再补解释段
  - 如果组件本身仍然发空，优先把组件内部做实，而不是回头补解释段落

### 已被否定的方向

- 只靠全局放大字号
- 只靠增加卡片高度
- 用没有解释意义的长短进度条装饰
- 在留白里继续堆字
- 把“给设计师/给 AI/给讲者自己看的注释语”直接展示给观众
- 用重复性的句子、标签或结论把同一页硬撑满
- 所有页都套同一个“标题 + 卡片堆叠”模板
- 过度拘泥于母版的双栏形态
- 拿到页面上来就改，不先判断这一页的讲演目标和视觉重心
- 为了显得满而硬塞内容
- 为了炫而加动画，却不帮助理解或讲述

### 母版结论

`s21` 现在已经是后续页面的重要母版来源，但它是“视觉母版 + 方法样板”，不是“固定模板”。

#### 真正固定下来的是这些

- 标题区的位置、字体、字号、分隔线
- 右上角页码的样式和位置
- 暖白色产品界面的配色、边框、圆角、阴影和留白节奏
- 主舞台容器、说明卡、动作板这几类容器的视觉语法
- 标题前缀并入标题的写法
- 如果页面前缀对观众有意义，应优先并入标题行，而不是单独显示为一行 kicker

#### 不固定的是这些

- 是否必须双栏
- 是否必须保留很多文字
- 双栏时左右宽度比例
- 右侧是否一定拆成“上概括卡 + 下动作板”
- 左侧是否一定是聊天窗
- 哪一块该做成动作型组件，哪一块该保持静态卡片

#### 什么时候该放大主舞台，什么时候该补组件

- 如果留白是一个完整的大矩形，并且主舞台本身有更多可展示内容，优先放大主舞台
- 如果主舞台已经达到合理尺寸，但旁侧或下侧仍有不规则空白，优先补一个和主叙事直接相关的动作型组件
- 如果只是为了“占地方”而新增文字卡，通常是错的
- 新增内容前先问：这块空白更适合被“窗口长大”吃掉，还是更适合放一个扫描、筛选、命中、追溯、流转之类的示例模块

#### `s21` 不该被机械复制

- 不要把后续页面都改成 `s21` 这张的长相。
- 要复制的是以下能力：
  - 精确对齐关键卡片
  - 把空说明卡改造成动作板
  - 让被放大的组件内部也同步变饱满
  - 用更高级、更产品化的方式组织标题、页码、边框、色彩
- 不必复制的是：
  - 一定是聊天窗
  - 一定是 `Claude` 白色风格
  - 一定是“上概括卡 + 下动作板”的右列结构

#### 关于对齐，要怎么理解

- 先对齐页面级的统一边距、标题区、页码区
- 再对齐真正对应的“收束模块”或“逻辑对应模块”，不要只盯整列外框
- 如果某个组件为了对齐被迫拉高，必须同步让它内部内容变饱满；否则应改结构，而不是继续拉伸

#### 在 `s21` 里，真正要对齐的是什么

- 不是抽象的“左右两列底边”
- 而是：
  - 左列底部两张并列说明卡的下底边
  - 对齐右列最下面“第二步：定向筛选”卡的下底边

#### `s21` 里已经证明有效的方法

- 右侧文字不多时，优先改成动作型组件板
- 如果动作板被拉高，必须同步把内部内容做满
- 例如：
  - 把翻译演示从 `4→4` 改成 `8→8`
  - 把筛选前后的关键词胶囊放大
- 让米黄色 demo 框收在白色总板内部

#### `s25` 这次暴露出的新规则

- 识别到左侧主卡底部发空后，不要第一反应就是补三张说明卡
- 要先估算空白区是不是更适合：
  - 把上面的对话窗口继续做大
  - 调整汇总表样式，让它自然吃掉更多高度
  - 或者补一个和“读表/统一口径/异常识别”直接相关的动画组件
- 任何补位动作都应该先判断空白大小和形状，再选方法，而不是生硬加内容
- 如果留白是横向长条，不要默认再补一排小卡；优先考虑：
  - 让上方主舞台继续长大
  - 或把这块做成横向流程/汇入/归并类组件
- 放大字号可以作为解决留白的方法之一，但必须和容器一起协调放大，不能只放大文字
- 放大字号时要同时检查：
  - 文字与卡片内边距是否仍然协调
  - 外框尺寸是否也随之调整
  - 关键文案是否会因此折行
  - 例如 `锦方月报` 这类标签，不应因为放大字号被挤成两行
- 动效组件不允许同质化复用：
  - 即使两个组件都承担“填补留白”的任务，也不应都做成同一种横向流线或同一种箭头桥
  - 要根据内容语义切换表现语言
  - 例如：
    - “6 份月报汇成 1 张表”更适合文档汇入/归并带
    - “字段别名归一”更适合归一引擎/核心中枢/映射台
- 后续判断某块留白该怎么补时，先问两个问题：
  - 这一块更像“信息流入”还是“规则处理中枢”？
  - 它和页面上已有组件是否会因为表现方式太像而显得思路固化？

#### `s25` 已验证有效的两种不同组件语言

- 左下角横向留白：
  - 适合做“文档汇入一张汇总表”的归并带
  - 优先通过增大外框、增大标签字号、保持单行标签来做满，而不是再塞解释卡
- 右栏中部留白：
  - 不适合继续复用左下角那种横向流线
  - 更适合做“字段别名归一”的中枢型组件，例如归一引擎、映射台、规则核心
  - 这样同一页内部的两个动画组件才有语义区别，也更有新鲜感

#### 这次全局修改新补充的硬规则

- 页面里不允许出现“注释型文字”“备注型文字”“设计者视角文字”
  - 一切类似“先让观众……”“这里是……”“下面要……”的字句，都必须从页面删除；若有必要，只能写进代码注释
- 页面里不允许为了排版规整而重复同一层信息
  - 一侧已经完整讲清的原因、判断或结论，另一侧不得改写后再次出现
- 当一页内容少但必须独立成页时，默认处理顺序不是“补文字”，而是：
  1. 提高真正重要文字的字号与分量
  2. 判断留白的形状与面积
  3. 设计一个与主叙事直接相关的、有意义的动作型组件
  4. 用组件去解释、证明或收束，而不是继续堆段落
- 如果一句主句、三问、一个判断就已经成立，优先让它们变重，再给它们配一个超大组件；不要补出第二层、第三层说明文字
- 动效组件优先级应高于“补一排说明卡”
  - 只要留白足够大，就优先考虑把这块让给更大的拟真组件、扫描组件、流程组件、问答组件或映射组件
- 留白过大时的补救顺序应为：
  1. 先看主组件是否还能更贴近讲演意图
  2. 再看主组件内部能否更饱满
  3. 再看其他区块是否该增重来压缩容器
  4. 最后才考虑新增别的说明块
- 如果一页改完后仍显得空，优先反思“这页主组件是不是还没有真正替演讲者把重点说出来”
  - 不是组件越复杂越好，而是要更像这一页真正想让观众看见的那个动作、判断或转折
  - 如果主组件已经对，但内部仍空，就优先把组件内部做实，而不是回头补说明句
  - 如果主组件本身已经足够实，也可以通过让其他关键区块更密、更重，来压缩容器和边框的视觉空腔
- 文字量必须和容器大小匹配
  - 任何文字溢出、挤压、贴边、需要靠极小字号硬塞进容器的情况都不允许出现
  - 如果一个组件容不下这些字，优先删字、收句，或者直接删掉这个组件，不要为了保留组件而牺牲可读性
- 卡片、模块、边框之间出现重叠，是红线
  - 一旦出现卡片互相压住、遮挡、跨边界重叠、文字压到别的容器上，这一页就不算完成
  - 在重叠问题没有彻底消失前，不允许把该页视为可交付成果
- 不要再用“低级补位”处理大留白
  - 不能只是把卡片或空框拉大，内部内容却不跟着变实
  - 如果留白很大，应优先考虑更有表现力的动效组件、图标/插画组件、扫描/流程/拟真组件
  - 只有在组件内部内容同步变实、比例也协调时，才允许适度放大容器
- 新增的组件必须“挣到自己的位置”
  - 如果一个组件既没有帮助理解，也没有帮助证明，更没有帮助收束，就应该直接删掉
  - 不要因为页面下方还有空间，就默认再补一套桥接组件、证明组件或说明组件
- 如果卡片内部使用 CSS grid，必须先核对“显式行数”与“实际子元素数量”是否匹配
  - 行数写少会制造隐式行，把正文或组件意外塞进弹性轨道，直接造成大块空白、错位甚至重叠
  - 卡片里的弹性高度应优先让主组件吃掉，不要让普通段落错误地占据 `1fr` 轨道

#### `s02 / s12 / s17` 这轮修改已经验证的经验

- `s02` 开场提问页：
  - 这类页不适合做成说明卡堆叠
  - 更好的做法是：保留少量真正重要的问题，再配一个足够大的“AI 已进入工作现场”主组件
  - 主组件要像演讲者想让观众立刻看见的现实场景，而不是抽象的左右信息板
- `s12` 时间窗口页：
  - 这类页不需要左右两栏各讲一遍原因
  - 三个理由说清后，剩余空间更适合交给一个大时间轴 / 窗口判断组件
  - 单栏重主句 + 大组件，比“双栏重复解释”更有效
- `s17` 概念页：
  - 概念页不能只靠定义句站住，要把概念本身变成可以观看的工作舞台
  - `Agent` 适合做成“收目标 → 自推进 → 交成果”的任务舞台
  - `Vibe Coding` 适合做成“说效果 → AI 生成 → 继续改”的造工具舞台
  - 底部收束区如果一句话就够，就不要再硬加桥接组件
  - 如果桥接组件字太多、容器太小、讲演价值又不强，应该直接删掉，而不是继续缩字硬塞

#### 最新补充的执行红线

- 母版本身也必须符合最终质量要求
  - 母版里如果存在重叠、拥挤、对齐失衡、容器发空等问题，后续页面会被一起带偏，必须先修母版认知再改页
- 不要把“改变卡片大小”当成留白处理的默认解
  - 这通常只是低级补位，不是完成度高的设计
  - 真正优先的是：更有趣、更有意义的动效组件，或图标/图形化组件
- “完成度高”比“先交一版”更重要
  - 如果页面还有重叠、严重留白、组件很弱、图形表达不足、明显像半成品，就继续改，不要急着展示
  - 给用户看的应尽量是已经收束、收敛、完成度较高的版本

#### 从这三页总结出的执行教训

- 拿到页面后，先问“这一页到底应该是问题页、判断页、概念页，还是工作现场页”
- 不要把“内容不多”误判成“必须继续补内容”
- 如果页面已经有一句足够重的话，优先围绕它做大主组件，而不是再加第二层解释
- 如果一个组件存在只是因为“版面这里空”，通常说明方向已经偏了
- 如果页面里某个小标题、本应一眼扫过的标签或判断句被迫换行，优先视为排版失败
  - 先调布局
  - 不行就缩字
  - 还不行就精简文案
- 后续章节标题行应尽量回到母版标题语法
  - 如果前缀对观众有意义，用 `title-prefix + title-sep + title-main` 合并进同一行
  - 不要再把可并入标题的前缀单独做成一行小字

## 11. 修改工作流

后续任何 AI 接手，建议严格按这个流程：

1. 先读 `README_AI接手指南.md`，再读 `ai-roadshow-s21-master.html` 和 `ai_roadshow_修改规格说明.md`
2. 如果不是只修单页，先完整过一遍该章节，再决定分页、合并、拆分和顺序
3. 优先用 `slide id` 定位具体页面；必要时再用 `visible page` 对齐用户口头页码
4. 如果已有截图或批量验收图，先看图；如果当前轮次不做逐页截图，也至少先看页面结构和现有实现，不要只扫文字
5. 明确这页承担的是：
   - 观点
   - 过程
   - 比较
   - 风险
   - 业务场景
   - 路线图
6. 再决定是改：
   - 结构
   - 字号
   - 卡片内部层级
   - 示例组件
   - 动效
7. 改完后做视觉复核
   - 默认可用截图
   - 如果当前轮次约定为“批量统一验收”，可先省去逐页截图，但不能省去最终整章或全稿回看

如果是全稿修改，还要多做两步：

8. 每完成一章，做一轮章节级回归
9. 继续下一章前，再重看一遍指导文件，避免要求漂移

## 12. 视觉复核方法

默认方式仍然是 headless Chrome 本地截图。  
如果当前轮次已经和用户约定“全部改完再统一验收”，可以省去逐页截图，但最终仍要做整章或全稿视觉回看。

常用方式：

```bash
'/Applications/Google Chrome.app/Contents/MacOS/Google Chrome' \
  --headless=new \
  --disable-gpu \
  --hide-scrollbars \
  --disable-background-networking \
  --disable-component-update \
  --no-first-run \
  --window-size=1600,900 \
  --virtual-time-budget=5000 \
  --screenshot='/Users/xpan/output/playwright/review-s39.png' \
  'file:///Users/xpan/Desktop/ai%20%E5%88%86%E4%BA%AB/ai-roadshow.html#39'
```

截图目录：

- `/Users/xpan/output/playwright/`

复核标准：

- 页面是否完整落在一屏内
- 是否真的把观众注意力带到了演讲者想强调的地方
- 主视觉是否明确
- 是否还有大面积无意义留白
- 是否只是堆字
- 微组件是否有解释意义
- 字和字、卡和卡是否对齐

## 13. 当前维护原则

- 只改 `ai-roadshow.html`
- `ai-roadshow-s21-master.html` 是参考母版，不是主实现文件
- 整章修改前，必须先重新建立章节级策略，不允许直接逐页开改
- 不另拆多文件版本
- 不破坏单文件可运行性
- 优先复用已有组件，不轻易再造一套新系统
- 每轮改完至少做一次视觉回看，不只看 DOM
- 做维护性重构时，优先抽配置、抽语义类、抽辅助函数，不借机改页面表达
- 后续优先让页面“更像演示现场”，不是“更像静态海报”

## 14. 给新 AI 的一句话总结

这不是普通网页，也不是普通 PPT。

它是一个单文件 HTML 路演系统，当前最重要的修改原则是：

- 保持 16:9 舞台感
- 不用大段文字填空
- 优先把抽象说明转成可观看、可讲述、可辅助理解的动作型组件
- 每改一页都要做视觉回看；若本轮约定批量验收，则至少在章节结束时统一回看
